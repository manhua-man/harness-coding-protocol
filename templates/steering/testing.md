# Testing Steering

> 适用于所有需要补测试、设计验证策略或解释测试边界的场景。

## Expectations

- 优先补最靠近行为边界的测试，而不是为了覆盖率堆低价值测试
- 先验证当前失败，再实现修复；避免“先改一遍再看是否好了”
- 说明没有跑到的测试、无法验证的环境依赖和剩余风险

## Preferred Order

1. 最小失败复现
2. 修复或实现
3. 运行相关测试
4. 说明回归范围与未覆盖部分

## Good Defaults

- 单元测试验证纯逻辑与边界条件
- 集成测试验证模块协作与 I/O 边界
- E2E / 手工验证只覆盖关键用户路径，不替代前两层
